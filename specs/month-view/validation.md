# Spec Validation: month-view

_Generated by /spec-validate — devil's advocate + architect review_

---

## Critical Issues (must resolve before building)

1. **Strip per entry vs. strip per unique discipline — direct contradiction.** The goal says "one strip per discipline" but Slice 2 says "render a strip per entry." These are different if one journal entry has multiple disciplines, or if two swim entries exist on the same day. Slice 3's cap and "+N" logic depends on this decision. Must be resolved before any code is written.

2. **`disciplines` field is title-case; `DISCIPLINE_ACCENT` keys are lowercase.** `DISCIPLINE_TAGS` stores `"Swim"`, `"Bike"`, etc. A direct `DISCIPLINE_ACCENT[discipline]` lookup returns `undefined` for every entry. Strips will silently not render. A normalization step (`.toLowerCase()` or a lookup map) is required but missing from the spec.

3. **`DISCIPLINE_ACCENT` doesn't cover all disciplines that can appear in journal entries.** Entries can be tagged `"Brick"`, `"Rest"`, `"General"`, `"Strength"`, `"Accessory"`, `"Soccer"` — none of these are in `DISCIPLINE_ACCENT`. No fallback color or "skip this discipline" rule is defined. Breaks silently.

4. **`useJournal` is not localStorage — it calls server actions.** The spec states "data comes from localStorage" but `hooks/useJournal.ts` already uses server actions with an async fetch and a `loading` state. Slice 2 must handle the loading flicker and the empty-on-first-paint case. This is not mentioned anywhere.

5. **Timezone handling is missing.** `localDateKeyInTz` lives as a private helper inside `WeekStrip.tsx`. MonthView will independently compute date strings from `new Date()`, causing DST-boundary mismatches (entries logged near midnight appear on the wrong cell). The helper must be extracted to a shared location before Slice 2.

6. **Week-start convention is unresolved.** The spec says "pick one" but WeekStrip already has a convention. MonthView must match it or users see misaligned columns. This is a must-decide, not a pick-any.

## Edge Cases to Add

- **Unknown/unmapped discipline** — define a fallback: skip the strip silently, or use a neutral color. Must be consistent with the dedup/per-entry decision.
- **Future-dated entries in the current month** — do they render? "Completed workouts" implies past only; add a filter for `date <= today`.
- **Duplicate discipline on same day** (two swim entries) — with "per discipline" rule this collapses to one strip; with "per entry" it's two. Spec must commit.
- **Strip ordering on multi-discipline days** — what order do strips stack? Chronological by entry, or discipline priority order? Unspecified causes inconsistency day-to-day.
- **Fixed grid height (6 rows vs. variable)** — if the grid is variable-height, the dashboard layout jumps month-to-month. Fix at 6 rows with trailing padding cells.
- **Today marker** — nearly every calendar highlights today's cell. Its omission is likely an oversight that will be flagged in code review.
- **Empty month** — no entries at all should still render the grid cleanly. Needs explicit acceptance criterion.
- **First-load loading state** — `useJournal` starts with `loading: true` and empty entries. Need a skeleton or a stable empty-grid render to avoid a flash.
- **Hydration mismatch** — `new Date()` on server vs. client can differ; Next.js will warn. Use `useEffect` or pass date server-side.
- **Color-only discipline encoding** — colorblind users cannot distinguish swim/run/bike strips. Consider a legend below the grid (even a small one) or a `title`/`aria-label` on each strip.

## Slice Amendments

**Add Slice 0 — Date utilities (before Slice 1)**
Extract `localDateKeyInTz` from `WeekStrip.tsx` to `lib/date.ts` (or `lib/utils.ts`). Both WeekStrip and MonthView consume it. This is a prerequisite for correct date grouping in Slice 2 and prevents TZ bug divergence between the two components.

Add `resolveDisciplineColor(tag: string): string | null` to `lib/constants.ts` (or `lib/utils.ts`). It handles lowercase normalization, non-core discipline mapping (Brick → `#FFD700`, etc.), and returns `null` for unmapped tags (Rest, General). Unit tested. Without this, Slice 2 will silently render no strips on most real data.

## Devil's Advocate Summary

## Weak Assumptions

1. **"Current month" is well-defined and stable.** The spec assumes a single `new Date()` call drives the view. What "current" means at 11:59 PM local vs. UTC is never specified. If grouping uses UTC dates but display uses local, entries logged near midnight will land on the wrong cell.

2. **`JournalEntry` has a reliable date field in `YYYY-MM-DD` form.** The spec says "group by date string (YYYY-MM-DD)" but never confirms what shape `JournalEntry.date` actually is. If it's a full ISO timestamp, a `Date`, or locale-formatted, the grouping logic breaks silently.

3. **`DISCIPLINE_ACCENT` covers every discipline that can appear in a journal entry.** WeekStrip has 7 discipline types beyond the core three. If journal entries can be tagged with any of those, `DISCIPLINE_ACCENT` lookups will return `undefined`.

4. **"Completed workouts" = "all journal entries."** The journal may contain reflections, rest-day notes, or mood-only entries with no discipline.

5. **One entry = one discipline.** The spec says "one strip per discipline" but also "render a strip per entry." These are different if a single entry tags multiple disciplines.

6. **Fixed cell height can absorb 3 strips + day number + padding.** No target cell height is given. On mobile 7-column grids, cells may be ~40px tall.

7. **Week start convention is a pick-one-and-move-on decision.** It isn't — WeekStrip already picked one.

8. **Month has ≤ 6 week rows.** True for Gregorian, but no one specified whether the grid is fixed at 6 rows or variable.

## Architect Summary

## Data Model Gaps
- `disciplines` is `string[]` with title-case values; `DISCIPLINE_ACCENT` keys are lowercase — lookup always returns `undefined`
- Non-core discipline tags (`"Brick"`, `"Rest"`, `"General"`) have no mapping in `DISCIPLINE_ACCENT`; `DISCIPLINE_CONFIG` in WeekStrip has them but it's component-private
- Multi-discipline single entries (brick day: `["Swim","Bike"]`) — spec doesn't say whether to render 2 strips for 1 entry
- No entry ordering rule for which disciplines win top-3 slots on multi-discipline days

## State & Integration Concerns
- `useJournal` is now server-action-backed with async fetch and `loading: true` initial state — the spec's "localStorage" description is outdated
- Each mounted `useJournal` is a separate state silo; adding an entry on `/athlete/journal` will not update the dashboard MonthView until navigation
- Filter by month first before sorting to avoid unnecessary work on full entry list

## Component & Module Boundaries
- Slices 1–3 are cleanly scoped to one new file; only `app/athlete/page.tsx` needs a new import
- `localDateKeyInTz` from `WeekStrip.tsx` is needed by MonthView — must be extracted to `lib/` before Slice 2 to avoid divergent TZ bugs
- MonthView should inherit the same layout wrapper/width as WeekStrip — not set its own

## Performance & UX Edge Cases
- First paint renders empty grid (loading state) — no skeleton specified
- DST transition months: entries near midnight may bucket to wrong day without shared TZ helper
- Brick/Rest/General entries will silently render no strip with current `DISCIPLINE_ACCENT` lookup
- Mobile: 7-column grid at ~360px = ~50px cells; 3 stacked strips + day number is very tight

## Missing Slices
- **Loading state** — `useJournal` is async; need explicit skeleton or stable empty-grid treatment
- **Date/TZ utility extraction** — extract `localDateKeyInTz` to shared `lib/` before Slice 2
- **Discipline normalization** — `resolveDisciplineColor(tag)` helper handling case + non-core disciplines, unit tested
- **Empty-month / first-load UX** — what does a new user with zero entries see?
- **Accessibility** — `aria-label` on cells, color-independent discipline encoding
- **E2E test** — add MonthView presence to existing smoke test in `e2e/smoke/navigation.spec.ts`

## Recommended Slice Order
1. Extract shared date helpers to `lib/date.ts`
2. Calendar shell (consuming shared helper)
3. Discipline resolution helper in `lib/constants.ts` (unit tested)
4. Workout strips (now trivial with helpers)
5. Multi-workout stacking + overflow (with explicit dedup rule)

## One Biggest Risk
The mismatch between `JournalEntry.disciplines: string[]` (title-case, multi-valued, includes `"Brick"`/`"Rest"`/`"General"`) and the spec's assumption of a clean 1:1 `DISCIPLINE_ACCENT` lookup. Slice 2 reads like a 30-minute task but will spawn hours of decisions: dedupe per day or per entry? Title-case or lower? Does Rest render? Does Brick borrow the gold from `DISCIPLINE_CONFIG`? Each decision has downstream effect on Slice 3's cap logic. Without a normalization contract written first, Slices 2 and 3 will ping-pong in rework.
